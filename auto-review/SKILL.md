# Auto Review: Multi-AI 설계 검증 워크플로우

이 스킬은 프로젝트 설계안을 작성한 뒤, OpenAI Codex(GPT)와 Google Gemini에게 교차 검증을 받아 품질을 높이는 자동화된 설계 리뷰 프로세스입니다.

## 인자

$ARGUMENTS 에 설계할 프로젝트의 주제/요구사항이 들어옵니다. 만약 비어있다면 사용자에게 설계할 프로젝트에 대해 질문하세요.

## 워크플로우

### Phase 1: 초기 설계 작성

1. 사용자의 요구사항($ARGUMENTS)을 분석합니다.
2. 아래 항목을 포함하는 **상세 설계 문서**를 작성합니다:
   - **프로젝트 개요**: 목표, 범위, 핵심 기능
   - **아키텍처 설계**: 시스템 구조, 컴포넌트 구성, 데이터 흐름
   - **기술 스택**: 선택한 기술과 선택 근거
   - **디렉토리 구조**: 프로젝트 파일/폴더 레이아웃
   - **핵심 모듈 설계**: 주요 모듈/클래스/함수의 인터페이스와 책임
   - **데이터 모델**: DB 스키마 또는 데이터 구조
   - **API 설계**: 엔드포인트, 요청/응답 형식 (해당 시)
   - **에러 처리 전략**
   - **테스트 전략**
3. 설계 문서를 사용자에게 보여줍니다.

### Phase 2: AI 교차 검증 루프

설계 문서가 준비되면 아래 반복 프로세스를 실행합니다:

#### Step 1: 리뷰 요청

OpenAI와 Gemini에게 **동시에** (병렬로) 리뷰를 요청합니다. 각 AI에게 아래 프롬프트를 전송합니다:

**리뷰 요청 프롬프트:**
```
당신은 시니어 소프트웨어 아키텍트입니다. 아래 설계 문서를 엄격하게 리뷰해주세요.

다음 관점에서 검토하고, 각 항목에 대해 구체적인 피드백을 제공하세요:
1. 아키텍처 적절성: 요구사항 대비 과/부족 설계 여부
2. 확장성/유지보수성: 향후 변경에 대한 유연성
3. 기술 스택 적합성: 선택한 기술의 적절성
4. 보안 취약점: 잠재적 보안 이슈
5. 성능 병목: 예상되는 성능 문제
6. 누락된 고려사항: 빠진 중요 요소
7. 모범 사례 준수: 업계 표준 패턴과의 일치 여부

응답 형식:
- [PASS] 문제없는 항목
- [ISSUE] 수정이 필요한 항목 (심각도: HIGH/MEDIUM/LOW)
- [SUGGESTION] 개선 제안 (선택적 반영)

마지막에 종합 판정을 내려주세요:
- APPROVED: 지적사항 없음, 이대로 진행 가능
- NEEDS_REVISION: 수정 필요 사항 있음

[설계 문서]
{설계 문서 전체 내용}
```

**OpenAI API 호출** (Bash 도구 사용):
```bash
curl -s https://api.openai.com/v1/responses \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -d '{
    "model": "o3",
    "input": "리뷰 프롬프트 + 설계 문서"
  }' | jq -r '.output[0].content[0].text'
```

**Gemini API 호출** (Bash 도구 사용):
```bash
curl -s "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-pro:generateContent?key=$GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "contents": [{"parts": [{"text": "리뷰 프롬프트 + 설계 문서"}]}]
  }' | jq -r '.candidates[0].content.parts[0].text'
```

#### Step 2: 피드백 분석 및 반영

1. 두 AI의 리뷰 결과를 종합합니다.
2. 사용자에게 리뷰 결과 요약을 보여줍니다:
   - 각 AI가 지적한 이슈 목록
   - 심각도 분류 (HIGH/MEDIUM/LOW)
   - 개선 제안 목록
3. **[ISSUE]로 표시된 항목**을 설계에 반영하여 수정합니다.
4. **[SUGGESTION]** 항목 중 타당한 것을 선별하여 반영합니다.
5. 수정된 설계 문서를 사용자에게 보여줍니다.

#### Step 3: 종합 판정 확인

- **두 AI 모두 APPROVED** → Phase 3으로 진행
- **하나라도 NEEDS_REVISION** → Step 1로 돌아가서 수정된 설계를 다시 검증
- **최대 반복 횟수: 5회** (무한 루프 방지). 5회 초과 시 현재 상태로 사용자에게 판단을 요청합니다.

### Phase 3: 최종안 확정 및 저장

모든 리뷰어가 승인하면:

1. 최종 설계 문서에 아래 메타데이터를 추가합니다:
   ```
   ## 리뷰 이력
   - 총 리뷰 라운드: N회
   - OpenAI 최종 판정: APPROVED
   - Gemini 최종 판정: APPROVED
   - 확정 일시: YYYY-MM-DD HH:MM
   ```

2. 프로젝트 루트(현재 작업 디렉토리)에 `DESIGN.md` 파일로 저장합니다.

3. 사용자에게 최종 결과를 보고합니다:
   ```
   ✅ 설계 검증 완료

   📄 최종 설계서: DESIGN.md
   🔄 총 리뷰 라운드: N회
   🤖 OpenAI: APPROVED
   🤖 Gemini: APPROVED

   코딩을 시작하려면 승인해주세요.
   ```

### Phase 4: 코딩 승인 대기

- 사용자에게 "코딩을 시작해도 될까요?"라고 질문합니다.
- 사용자가 승인하면 DESIGN.md를 기반으로 구현을 시작합니다.
- 사용자가 추가 수정을 요청하면 해당 부분을 수정 후 Phase 2를 다시 실행합니다.

## 중요 참고사항

- API 키가 환경변수에 설정되어 있어야 합니다: `OPENAI_API_KEY`, `GEMINI_API_KEY`
- API 키가 없으면 사용자에게 설정 방법을 안내하세요.
- 각 API 호출 시 에러가 발생하면 사용자에게 알리고, 해당 AI의 리뷰를 건너뛸지 물어보세요.
- 리뷰 프롬프트의 언어는 설계 문서의 언어에 맞춥니다 (한국어 설계 → 한국어 리뷰 요청).
- curl 명령에서 JSON 내 특수문자(따옴표, 줄바꿈 등)는 반드시 이스케이프 처리합니다. jq를 활용하여 JSON payload를 안전하게 구성하세요.
