# Project Report — 프로젝트 종료 보고서 생성 스킬

프로젝트 종료 시 **요약보고서**와 **상세보고서**를 `./doc/` 디렉토리에 체계적으로 저장합니다.

## 트리거 조건

- 사용자가 `/project-report`를 호출할 때
- 프로젝트 종료 시 보고서 작성이 필요할 때

## 실행 절차

### 1단계: 프로젝트 정보 수집

현재 작업 디렉토리(CWD)에서 프로젝트 정보를 파악합니다:

1. **프로젝트명 결정** (우선순위):
   - `package.json`의 `name` 필드
   - `pyproject.toml`의 `[project] name`
   - `Cargo.toml`의 `[package] name`
   - `go.mod`의 module 이름
   - `pom.xml`의 `<artifactId>`
   - 현재 디렉토리 이름 (`basename $PWD`)

2. **오늘 날짜**: `YYYY-MM-DD` 형식

3. **버전 번호 결정**:
   - `./doc/` 디렉토리에서 같은 프로젝트명+날짜+보고서 유형으로 된 기존 파일 검색
   - 패턴: `{프로젝트명}-{날짜}v*.md`
   - 기존 파일이 없으면 `v1`
   - 기존 파일이 있으면 가장 높은 버전 + 1
   - **요약/상세 보고서가 같은 버전을 공유**

### 2단계: 프로젝트 현황 분석

다음 소스에서 프로젝트 정보를 종합적으로 수집합니다:

1. **코드베이스 구조**: 디렉토리 구조, 주요 파일/모듈 파악
2. **Git 히스토리**: 커밋 로그, 기여자, 브랜치 현황
3. **기존 문서**: `./doc/` 내 설계 문서, README, CHANGELOG 등
4. **의존성**: `package.json`, `requirements.txt`, `Cargo.toml` 등에서 주요 의존성
5. **테스트**: 테스트 파일 존재 여부, 테스트 커버리지 (가능한 경우)
6. **대화 컨텍스트**: 현재 대화에서 논의된 프로젝트 관련 내용

사용자가 `/project-report` 뒤에 추가 인자를 제공하면 해당 내용도 반영합니다.

### 3단계: 요약보고서 생성

### 5단계: 결과 보고

저장 완료 후 다음을 출력합니다:

```
## 프로젝트 종료 보고서 생성 완료

| 보고서 | 파일 |
|--------|------|
| 요약 보고서 | `./doc/{프로젝트명}-summary-{날짜}v{버전}.md` |
| 상세 보고서 | `./doc/{프로젝트명}-detail-{날짜}v{버전}.md` |

- **프로젝트**: {프로젝트명}
- **버전**: v{버전}
- **이전 버전**: {있으면 목록, 없으면 "없음"}
```

## 사용 예시

```
/project-report                          # 현재 프로젝트의 종료 보고서 생성
/project-report MVP 완료 기준 보고서       # 추가 컨텍스트 포함
```

## 파일명 규칙

- 프로젝트명에서 `@`, `/` 등 특수문자는 제거하고 소문자로 통일
- 공백은 `-`로 치환
- 요약: `{프로젝트명}-summary-{날짜}v{버전}.md`
- 상세: `{프로젝트명}-detail-{날짜}v{버전}.md`
- 예: `my-project-summary-2026-03-09v1.md`, `my-project-detail-2026-03-09v1.md`

## 수집 시 주의사항

- **시크릿 제외**: `.env` 내용, API 키, 비밀번호, 토큰 등은 절대 보고서에 포함하지 않음
- 환경 변수 섹션에는 변수명과 설명만 기재, 실제 값은 `[REDACTED]`
- 기존 버전 파일은 절대 덮어쓰지 않음 (항상 새 버전으로 생성)
- `./doc/` 디렉토리가 없으면 자동 생성
- 프로젝트 정보가 부족하면 사용자에게 확인 후 작성
- 대화 컨텍스트와 코드베이스 분석 결과를 종합하여 최대한 풍부한 보고서 작성
